# Allocation Remediation Plan

## Objectives
- Restore zero-allocation per-row evaluation for model matrix rows without metaprogramming.
- Keep execution type-stable across LM/GLM/LMM/GLMM and interactions/functions.
- Preserve current API and maintain performance or better.

## Current Status (from test/allocation_results.csv)
- Continuous terms allocate ~16 bytes; functions ~32 bytes; complex interactions higher.
- Pure categorical cases are already 0 bytes.
- File: `test/allocation_results.csv` generated by `test/allocation_survey.jl`.

## Root Causes (code review)
- Continuous: runtime `Symbol`-based access in CompleteFormula execution (e.g., `get_data_value_specialized(data, ::Symbol, ...)`).
- Functions: some paths evaluate arguments via `Symbol` lookups instead of typed sources.
- Interactions: value retrieval accepts `Symbol`/`Int`/scratch inputs; continuous branches do not consistently leverage type parameters.
- Categorical: specialized path stores `column::Symbol` and reads via `getproperty(input_data, column)` (type-unstable), though simple cases still hit 0 bytes.

## Strategy & Phases

### Phase 1 — Continuous (fast win)
- Replace CompleteFormula continuous execution with the step1 typed executor that uses `Val{col}` dispatch.
  - In `src/step4_interactions.jl`, route continuous stage through `execute_operation!(data.continuous, op.continuous, ...)` which calls `get_data_value_type_stable(input_data, Val{col}(), ...)`.
- Remove/demote `execute_complete_continuous_operations!` to avoid accidental use.
- Expected: all LM/GLM/LMM/GLMM cases without functions drop from 16 bytes to 0.

### Phase 2 — Categorical stabilization
- Specialize categorical data by column at the type level:
  - Redefine `SpecializedCategoricalData{N,Positions,Column}` and access levels via `getproperty(input_data, Val{Column}())` + `extract_level_code_zero_alloc`.
  - Update `analyze_categorical_operations` to construct typed tuples.
- Expected: keep 0 bytes for categorical; make path robust for interactions/functions.

### Phase 3 — Functions (typed inputs)
- Ensure function inputs are typed sources, not `Symbol`s:
  - `src/step3_functions.jl`: use/extend `get_input_value_zero_alloc` to prefer `Val{Column}` and `ScratchPosition{P}`; ensure decomposition emits typed input sources.
  - `src/step4_interactions.jl`: in `get_value_from_source` and FunctionEvaluator branches, swap `get_data_value_specialized` for `get_data_value_type_stable(input, Val{Column}(), ...)` when `ContinuousEvaluator{Column}` is known.
- Expected: function rows drop from ~32 bytes to 0.

### Phase 4 — Interactions (end-to-end)
- Harden interaction retrieval to avoid `Symbol` access:
  - Keep index patterns as tuples; keep pre-evals as tuples.
  - For continuous components, route reads through component types (`ContinuousEvaluator{Column}`) or typed sources.
  - For categorical components, rely on typed `SpecializedCategoricalData{...,Column}`.
- When the typed path passes all benchmarks, remove the metaprogramming fallback in `src/generated_function_interactions.jl`.

### Phase 5 — Tests, Survey, CI
- Microchecks (0 allocations):
  - Continuous-only, categorical-only, mixed, functions, interactions (cat×cat, cont×cat, func×cat).
  - Use `@allocated compiled(buf, data, 1) == 0` for representative LM/GLM/LMM/GLMM.
- Survey:
  - Run `julia --project=test test/allocation_survey.jl` and examine `test/allocation_results.csv`.
  - Guardrails: fail CI if any “core” rows (simple, mixed, interaction, function) show `memory_bytes > 0` (allow explicit exceptions if needed).
- Optional CI: GitHub Actions step to instantiate + run survey and grep for nonzero allocations.

## Affected Files (primary)
- `src/step4_interactions.jl`: continuous execution, function/interaction value retrieval.
- `src/step2_categorical_support.jl`: define `SpecializedCategoricalData{...,Column}`; update execution.
- `src/step3_functions.jl`: ensure typed input sources and zero-alloc input retrieval.
- `src/compile_term.jl`: ensure `ContinuousEvaluator{Column}` and typed propagation are preserved.
- Remove later: `src/generated_function_interactions.jl` (post-validation).

## Risks & Mitigations
- Risk: accidental dynamic dispatch reintroduced via helper paths → Mitigate by centralizing Val-based access and adding micro allocation asserts.
- Risk: MixedModels schema differences → Covered by survey; add targeted LMM/GLMM micro tests.
- Risk: Performance regressions → Track `time_ns` in survey; compare before/after.
